# Research: Adversarial Review of M2 Proposal

> Second-pass research. Each finding was generated by an agent tasked with *challenging* round-one conclusions. Treat claims with extra scrutiny — they are designed to find weaknesses, and some overreach.

## 1. The "No Published System Evolves Specifications" Claim Is Overstated

Round one (files 01, 07) asserted: "No published system has demonstrated LLM-driven evolution of a structured specification document that is then used to construct an implementation." This is wrong in spirit and partially wrong in letter.

### Systems that evolve structured text artifacts via LLM

**PromptBreeder** (DeepMind, ICML 2024, arXiv 2309.16797). Evolves both task-prompts and mutation-prompts. The mutation strategy itself is under evolutionary pressure — self-referential self-improvement. This is what the M2 proposal defers to M3+, but PromptBreeder does it from the start. The scale difference (short prompts vs. multi-page specs) is real, but the mechanism is identical.

**VisionForge** (2025, arXiv 2503.03967). Evolves model behavior specifications — structured natural language documents defining LLM behavior — through self-play. The LLM simulates adversarial users, identifies failure triggers via conversation backtracing, then rewrites the specification. This is the closest published analog to Cambrian M2: a structured behavioral specification mutates based on fitness signals, and model behavior is the phenotype.

**TextGrad** (Stanford, Nature 2024, arXiv 2406.07496). Treats any text as an optimizable variable with LLM-generated "textual gradients." Provides directional feedback rather than random mutation — potentially more sample-efficient for spec refinement.

**DSPy** (Stanford, ICLR 2024, arXiv 2310.03714). Typed module signatures are structured specifications; its optimizer evolves implementations to match them. Module signatures compose via type checking — decomposed yet coherent.

**EvoPrompt** (ICLR 2024, arXiv 2309.08532). Applies genetic algorithm / differential evolution to prompt populations. Evolutionary operators on structured text.

**OPRO** (DeepMind, 2024, arXiv 2309.03409). Uses the LLM itself as the optimizer over text, guided by past performance.

**GitHub Spec-Kit** (GitHub, 2025-2026). Industry toolkit for specification-driven development where specs are living, version-controlled artifacts driving LLM code generation. The community is actively debating automated spec evolution.

### What survives this challenge

The M2 claim must be narrowed. What IS novel:

1. **Genotype-phenotype indirection**: Most systems evolve the artifact that IS the output (prompts → prompt quality, code → code quality). Cambrian evolves a specification that a *separate constructor* interprets to produce an implementation. The spec is the genotype; the generated codebase is the phenotype. This indirection is real, published nowhere else at this scale, and architecturally important.

2. **Recursive self-reproduction as fitness test**: The offspring must read its own spec and reproduce. This recursive viability requirement is unlike prompt optimization or code evolution.

3. **Scale**: PromptBreeder evolves paragraphs. VisionForge evolves behavioral rules. Cambrian M2 would evolve a multi-thousand-word technical specification. No published system operates at this document scale.

**Update needed**: Revise the thesis to claim indirection + self-reproduction + scale, not uniqueness of text evolution.

## 2. Monolithic Spec Will Evolve Toward Entanglement

### The biology case against monolithic genomes

**Supergene degeneration** (PMC 4780387): Tightly linked loci with suppressed recombination (the biological analog of a monolithic spec) suffer genetic degeneration, reduced effective population size, and Muller's ratchet — irreversible accumulation of harmful mutations. Without independent recombination of sections, bad mutations in one section hitchhike on beneficial mutations in another.

**Linkage drag**: In a monolithic spec, a beneficial change to one section paired with a degraded change to another will be selected if net fitness is positive. Campaign-level fitness (viability_rate, fitness_mean) is a whole-genome measure that *cannot* distinguish which section contributed. Over time, rarely-tested sections degrade.

**Clune on modularity** (Royal Society, 2013): Modularity evolves only when there is a connection cost. Without this pressure, systems remain monolithic and entangled. The M2 spec has NO modularity pressure — no penalty for cross-section dependencies. Prediction: the spec will evolve toward increasing entanglement, making Type 2 (transplant) mutations progressively harder to perform without breakage.

### The computational case for decomposition

**AlphaEvolve's EVOLVE-BLOCK decomposition**: Specific blocks mutate independently within a shared scaffold. Cross-reference integrity is maintained by the scaffold (unannotated code) serving as the fixed reference frame, plus the evaluator catching breakage. This is strictly more flexible than a monolithic approach — you can mark exactly which sections are evolvable and which are context.

**DSPy's typed module composition**: Modules with typed signatures are optimized independently. Type checking enforces interface contracts between modules — achieving cross-module consistency without requiring monolithic document coherence.

**Modular CGP** (GECCO 2024-2025): Decomposed genomes with module reuse outperform monolithic genomes on complex problems because modularity enables independent optimization, module reuse, and reduced search space per mutation.

### What this means for M2

The monolithic spec is not wrong for Stage 1 — it's simpler and the spec is small enough that entanglement hasn't had time to accumulate. But the proposal needs to:

1. **Add section-level fitness attribution**: After a mutation, diff the spec and attribute fitness changes to specific sections. Which section caused the improvement or regression?
2. **Monitor entanglement**: Track how many sections each mutation touches. If mutations increasingly need to touch multiple sections to achieve any improvement, entanglement is growing.
3. **Plan for decomposition**: If entanglement becomes a problem, the spec should decompose into typed modules (like DSPy signatures) with explicit interfaces between sections.

## 3. Grammar-Constrained Mutation Outperforms Free-Form

**SBMaOG3P** (MDPI Algorithms, 2024): Maps LLM-generated code to a BNF grammar, then applies grammar-obeying genetic operators. Grammar constraint improves the rate of valid program generation vs. unconstrained LLM output.

**Hybrid GA-LLM** (arXiv 2506.07483, 2025): GA maintains structural integrity while LLM provides creative variation. Separating structural and semantic concerns is more efficient than using the LLM for both.

**Constrained decoding** (growing body of work, 2024-2025): Constraining LLM token generation at decode time to obey schemas/grammars guarantees structural validity — eliminating post-hoc screening entirely.

**Implication**: The Cambrian spec has implicit structure: markdown heading hierarchy, MUST/MAY keywords, table schemas, JSON code blocks, the frozen invariant section. A formal grammar (even a lightweight one — headings, required sections, keyword constraints) would make many incoherent mutations structurally impossible rather than caught by the screener. This directly intersects with the recommendation to drop the dual-model screener (see §5 below).

**Recommendation**: Define a lightweight spec schema (required sections, heading hierarchy, keyword vocabulary) for Type 1 and Type 2 mutations. Reserve free-form mutation for Type 3 (restructuring), where breaking structure is the point.

## 4. MAP-Elites Is Wrong for Cambrian's Budget

### The budget problem

MAP-Elites with 4 behavioral dimensions and 5 bins per dimension creates a 625-cell archive. Each evaluation costs real money ($0.08–$1.00 per campaign from lab journal data). Filling even 10% of the archive costs $5–$63 *before the archive becomes useful for selection*. The quality-diversity literature confirms this concern: MAP-Elites needs hundreds or thousands of evaluations for problems with expensive evaluations.

### Better alternatives for expensive evaluations

**BOP-Elites** (Kent & Branke, 2023, arXiv 2307.09326): Bayesian Optimization of Elites. Models fitness AND behavioral descriptors with Gaussian Process surrogates. Designed for "problems with expensive black-box fitness and behavior functions" with "excellent final performance after a relatively small number of samples."

**SAIL** — Surrogate-Assisted Illumination (Gaier et al., 2018, arXiv 1806.05865): Runs MAP-Elites on a GP surrogate (cheap), evaluates only the most promising real candidates. Achieves MAP-Elites quality with "several orders of magnitude fewer evaluations."

**BEACON** (2024, arXiv 2406.03616): Bayesian novelty search with multi-output GPs. "Significantly outperforms existing novelty search baselines under tight evaluation budgets."

### The deeper question

The M2 proposal justifies MAP-Elites by saying "we don't know which region of the fitness space matters for the next task." But M2 Stage 1 has ONE task: improve viability rate for self-reproduction. There is no task distribution to hedge against. Plain Bayesian optimization is state-of-the-art for single-objective expensive optimization with 50–100 evaluations.

**Recommendation — staged algorithm:**
- Stage 1 (~20–30 campaigns): Single-objective BO. Maximize viability rate. Simple, cheap, proven.
- Stage 2 (~50–100 campaigns): BOP-Elites. Add behavioral dimensions when you have budget to fill them.
- Stage 3 (200+ campaigns): Full MAP-Elites. Only when the archive can be meaningfully populated.

This matches the finding from file 07 (A-Evolve) that single-lineage evolution is a viable starting point.

## 5. Drop the Dual-Model Screener

### LLM-as-judge is hackable under selection pressure

**"One Token to Fool LLM-as-a-Judge"** (2025, arXiv 2507.08794): "Superficial inputs termed 'master keys' — such as non-word symbols or generic reasoning openers — can consistently elicit false positive rewards. False positive rates can reach 80%." This affects leading proprietary systems including Claude.

The M2 screener (Sonnet checking spec coherence) is structurally vulnerable: LLMs overfit on surface features (formatting, length, structure) rather than semantic coherence. Under selection pressure, the mutation loop *will* find specs that look coherent to Sonnet without being coherent. This is Adversarial Goodhart's Law — when the optimizer knows the metric, the metric loses validity.

### Evidence that single-model works

**ThetaEvolve** (arXiv 2511.23473): Dropped the dual model entirely and achieved SOTA results by using a single LLM with a larger population (10,000 vs 70 programs). Budget spent on the screener is better spent on more evaluations.

### The Test Rig is the correct judge

The M2 proposal already has a deterministic, immutable, isolated evaluator — the Test Rig. It cannot be gamed by surface features. It runs real builds, real tests, real health checks. The screener is redundant if the Test Rig exists.

**Recommendation**: Opus generates mutations. Every mutation goes directly to a short campaign (2–3 generations for quick screening, full 5 for promising ones). The Test Rig IS the judge. Keep deterministic string-comparison checks for frozen invariant violations — this is not an LLM judgment, it's `grep`.

**Budget impact**: Eliminating Sonnet screener calls saves ~$0.01–$0.05 per mutation. Small per-call, but compounds over hundreds of mutations. More importantly, it removes a gameable surface from the selection loop.

## 6. Container Isolation Is Broken

### The threat

The organism's `entry.build`, `entry.test`, and `entry.start` execute with full filesystem access inside the same container where the Test Rig runs. Under evolutionary pressure, a spec could describe code that:
- Writes a fake `viability-report.json` directly
- Modifies `/test-rig/` code between pipeline stages
- Manipulates the shared workspace to influence Test Rig behavior
- Reads the Test Rig source to reverse-engineer what passes

This is directly analogous to documented attacks in evolutionary computation where organisms deleted target files to achieve perfect fitness scores.

### The invariants don't prevent this

The 5 frozen invariants prevent Prime from modifying the spec or performing git operations. They do NOT prevent the *generated code* (the organism) from:
- Writing to arbitrary filesystem paths during `entry.build` or `entry.test`
- Reading the Test Rig's source code to understand what's being evaluated
- Manipulating the environment between pipeline stages

The invariants are constraints on *what Prime writes in the spec*. The organism — the code Prime generates — has no such constraints once it executes.

### How serious is this?

**High severity, easy fix.** This is the most actionable finding in this review. The fix is well-understood:
1. Run organism commands in a separate process namespace
2. Mount `/test-rig/` read-only (or don't mount it at all)
3. Run organism commands as non-root user
4. Run the Test Rig's evaluation stage with a fresh copy of the workspace, not the one the organism may have modified

**Cambrian relevance**: The current architecture (BOOTSTRAP-SPEC-002) may already address some of this — but the M2 proposal needs to verify that evolutionary pressure on the spec can't produce organisms that game the Test Rig through filesystem manipulation.

## 7. Self-Referential Fitness Dimensions

### The problem

Three of the 15 fitness dimensions are self-referential: `test_count`, `test_pass_rate`, and `test_coverage`. The organism generates its own test suite. Under selection pressure, a spec will evolve to describe code with:
- 500 `assert True` tests (maximizing `test_count` and `test_pass_rate`)
- Empty `requirements.txt` (minimizing `build_duration_ms`)
- Trivial implementations that pass trivial tests

Research shows 30–32% of LLM-generated solutions that pass their own tests still have fundamental logical errors (EvalPlus, arXiv 2305.01210).

### Mitigation

Add test-quality metrics to the fitness vector:
- **Mutation score**: Run mutation testing to check if tests catch injected faults
- **Assertion density**: Meaningful assertions per test (not `assert True`)
- **Coverage of spec requirements**: Does the test suite cover the behavioral contracts in the spec?

The Red Queen tiers partially address this (Tier 1 adds behavioral contracts), but the organism's own `test_pass_rate` should be discounted or replaced by the Test Rig's external evaluation.

## 8. Replace Fixed Neutral Drift with Temperature Annealing

### The problem

The M2 proposal triggers neutral drift every 10 campaigns for 2 campaigns — 20% of total budget spent without selection pressure. The M2 success experiment is 10 campaigns. Spending 2 on neutral drift means 20% of the experiment generates no directional signal.

Neutral drift's theoretical justification comes from population genetics with large populations over long timescales. In populations of 10–50 individuals (Cambrian's likely archive size), neutral drift primarily increases variance without the population to buffer against it.

### Better alternative

**CMA-MAE** (Covariance Matrix Adaptation MAP-Annealing): Temperature annealing that smoothly interpolates between uniform acceptance (exploration) and elitist selection (exploitation). The temperature parameter decays over time, providing exploration when the archive is sparse and exploitation when it's populated. No manual scheduling, no wasted budget.

If keeping MAP-Elites (Stage 3), use CMA-MAE's temperature schedule rather than fixed neutral drift periods. If using BO/BOP-Elites (Stage 1–2), the acquisition function's explore/exploit balance replaces neutral drift entirely.

## 9. Adaptive Test Generation (Lightweight Co-Evolution)

### The gap

The Red Queen tiers are a fixed curriculum, not co-evolution. If the system plateaus at Tier 1, Tier 2 tests are irrelevant. The tiers cannot target specific recurring failure modes.

### The evidence

**POET** (Wang & Lehman, 2019, arXiv 1901.01753): Co-evolves environments that are slightly beyond current capability. Creates a natural curriculum rather than fixed gates.

**Absolute Zero Reasoner** (NeurIPS 2025 Spotlight, arXiv 2505.03335): Generates its own training tasks via self-play with a code executor as verifier. No external data. Achieves SOTA on coding and math reasoning, "outperforming models that rely on tens of thousands of human-curated examples." Proves test co-evolution works for code generation specifically.

### Recommendation

Keep Tier 0–3 as immutable floors. Add "Tier X" (adaptive): after each failed campaign, extract the failure mode, use an LLM to generate 3–5 targeted test cases probing that failure, add them to the next campaign's evaluation. The adaptive tests probe *how* to satisfy tiers, not *what* to satisfy.

**Caution**: Co-evolved tests can drift toward testing irrelevant edge cases. The fixed tiers anchor what "viable" means; adaptive tests explore failure surfaces within those tiers.

## 10. The DGM Motivation Is Overstated (Minor)

The M2 proposal repeatedly cites Darwin Godel Machine as the cautionary tale for self-evaluation. However, the DGM paper (arXiv 2505.22954) actually found zero cases of reward hacking in controlled experiments. The self-improvement worked — the DGM improved its own code and tests across iterations.

This doesn't undermine the M2 safety architecture (external evaluation is still a good idea), but the *justification* should cite Goodhart's Law and specification gaming more broadly, not DGM specifically. The DGM failure mode is a theoretical risk, not a demonstrated one.

## Cross-System Patterns (Round 2)

1. **The uniqueness claim must be scoped precisely.** Cambrian's novelty is genotype-phenotype indirection + self-reproduction + spec scale. It is NOT "the first system to evolve text via LLM." PromptBreeder, VisionForge, TextGrad, DSPy, EvoPrompt, and OPRO all evolve text. Be precise about what's new.

2. **Budget is the binding constraint.** Every architectural choice — MAP-Elites, dual-model, neutral drift, 4D behavioral space — was designed for a world with abundant evaluations. Cambrian lives in a world where each evaluation costs real money. Stage the complexity to match the budget: BO → BOP-Elites → MAP-Elites; short campaigns for screening → full campaigns for promising mutations.

3. **The container isolation gap is the highest-priority fix.** Under evolutionary pressure, organisms WILL find filesystem-based attacks on the Test Rig if the execution boundary is permeable. This is not speculative — it's documented in the evolutionary computation literature and it's the one thing that could make the entire safety architecture fail.

4. **Grammar-constrained mutation is strictly better than free-form + post-hoc screening** for Type 1 and Type 2 mutations. If the spec has structure (and it does), enforce that structure at mutation time rather than filtering invalid mutations after the fact.

5. **Section-level fitness attribution is the precondition for evolvability.** Without it, beneficial mutations in one section will drag harmful mutations in adjacent sections into the archive. This is linkage drag — the most common failure mode of monolithic genomes — and the M2 proposal has no defense against it.
