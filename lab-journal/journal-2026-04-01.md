## Phase 1: Bug Fixes

Resumed from the previous session's spec compliance audit (journal-2026-03-31g). Five P1 beads were ready with no blockers. All five were fixed in a single commit.

### cambrian-2sg — CAMBRIAN_TOKEN_BUDGET misused as max_tokens

**File:** `supervisor/prime_runner.py:27`

`_MAX_TOKENS = int(os.environ.get("CAMBRIAN_TOKEN_BUDGET", "32768"))` was being passed as `max_tokens` on every LLM call. Per CAMBRIAN-SPEC-005 §Configuration (line 559), `CAMBRIAN_TOKEN_BUDGET` is a *cumulative* budget across the entire run (0 = unlimited); it has nothing to do with the per-call `max_tokens` parameter. Cumulative budget tracking belongs to the Meta-Monitor (M2 Stage 2+, deferred).

**Fix:** Replaced with a hardcoded `_PER_CALL_MAX_TOKENS = 32768`. The env var is no longer read in prime_runner.

### cambrian-5tym — Model escalation missing on retries

**File:** `supervisor/prime_runner.py:209`

`CAMBRIAN_ESCALATION_MODEL` was never read. The same model was used for all retry attempts. Per spec §Generation Loop (line 258): "Model: `CAMBRIAN_MODEL` on the first attempt (`retry_count == 0`); `CAMBRIAN_ESCALATION_MODEL` (default `claude-opus-4-6`) on any retry (`retry_count >= 1`)."

**Fix:** Added `_ESCALATION_MODEL = os.environ.get("CAMBRIAN_ESCALATION_MODEL", "claude-opus-4-6")`. Inside the loop: `call_model = _ESCALATION_MODEL if attempt > 1 else model`. The `stream()` call and the manifest's `producer-model` field both now use `call_model`.

### cambrian-7djj — entry.build uses pip instead of uv pip

**File:** `supervisor/prime_runner.py:125`

The manifest template had `"build": "pip install -r requirements.txt"`. The spec (line 142) and the whole project's tooling policy both require `uv pip`.

**Fix:** One-word change to `"uv pip install -r requirements.txt"`.

### cambrian-pom1 — Rollback tag suffixing violates spec

**File:** `supervisor/git_ops.py:83`

`rollback()` was checking for existing tags matching `gen-N-failed*` and appending a numeric suffix (`gen-N-failed-2`, etc.). Per BOOTSTRAP-SPEC-002 §Git Operations (line 757): "Each retry gets a new generation number, so tags are always `gen-N-failed` — never suffixed." If a tag already exists (which indicates a logic bug upstream), the spec says to log a warning and overwrite.

**Fix:** Removed the tag-listing and suffix logic. Always use `gen-N-failed`. Use `git tag -fa` (force annotated) to overwrite. Log a `rollback_tag_exists` warning if the tag was already present.

### cambrian-h4j3 — git add -A stages entire artifacts repo

**File:** `supervisor/git_ops.py:59`, `supervisor/supervisor.py:201`

`create_generation_branch()` ran `git add -A` with no path argument, staging *all* files in the artifacts repo. The spec pseudocode (BOOTSTRAP-SPEC-002 line 261) explicitly passes `artifact_path` to scope the staging to one generation's directory.

**Fix:** Added `artifact_path: str = "."` parameter to `create_generation_branch()`. Passes it to `git("add", "-A", artifact_path)`. Updated the caller in `supervisor.py:201` to pass `artifact_rel` (already available in scope).

## Key Observations

1. **prime_runner.py has no unit tests.** All five bugs existed undetected because the module has zero test coverage. The supervisor and lifecycle tests mock `git_ops` entirely and never import `prime_runner`. This is the most-changed module in M2 infrastructure and the most dangerous to have untested. Opening a bead for this would be worthwhile before the next M2 campaign.

2. **Two of the five bugs (cambrian-2sg, cambrian-5tym) trace to the same root cause as the streaming bugs in the previous session:** prime_runner.py was written from intent rather than from the spec's LLM Integration and Configuration sections. The pattern is now clear — any new M2 infrastructure module needs a spec cross-check pass before merging.
