## Phase 1: Smoke-Import Pre-Start Check (cambrian-tbnt)

**Motivation:** Gen-16 failed at the `start` stage with `ModuleNotFoundError: No module named 'src'`. The root cause was `python src/prime.py` vs `python -m src.prime` — a `sys.path` difference that tests can't catch because pytest adds CWD to `sys.path` automatically. The fix (`python -m src.prime`) was added to the spec in v0.14.2, but structural failures like this can still occur if the LLM mis-generates the import style. An early check costs <1s and surfaces the error before the process launch.

**Implementation:** Added `run_import_check(entry)` to `test-rig/test_rig.py`:
- Extracts the module name from `-m <module>` in `entry.start` using `shlex.split()`
- Runs `sys.executable -c "import <module>"` in WORKSPACE
- Returns `passed=True` immediately if no `-m` flag (fallback: start command doesn't use module form)
- Reports failure under the `start` stage — no schema change

The check runs as the first thing in Stage 4 (start), before `start_process()`. If it fails, the diagnostics include the stderr from the failed import, making the root cause immediately visible.

**Tests added:** `TestRunImportCheck` in `test-rig/test_test_rig.py` — 5 tests covering: no `-m` flag, no start command, importable module, missing module, `python3` variant.

### Gen-17 Analysis — structlog misuse

`prime.py` called:
```python
log.info("event", event="prime_starting")
```

structlog's `.info()` first positional arg IS the `event` key. Passing it as a positional arg AND a keyword arg gives `TypeError: got multiple values for argument 'event'`. The correct call is `log.info("prime_starting")` or `log.info("event", type="prime_starting")`. This is a common mistake — structlog's API is non-obvious vs the standard library's `logging.info()`.

**Manifest:** `"start": "python src/prime.py"` ← wrong form (see discovery below).

---

### Gen-18 Analysis — silent loop failure → exit 0

`prime.py` started the TCP site then awaited a background task (`run_generation_loop()`). The task called `from src.loop import generation_loop` (lazy import inside a function). Because the start command was `python src/prime.py`, `/workspace/src` was on `sys.path` — so `import src.loop` failed with `ModuleNotFoundError: No module named 'src'`. The exception was caught by `except Exception as exc`, cleanup ran, server stopped, process exited with code 0.

Timeline: TCP site bound briefly → task failed → cleanup → exit 0 — all within ~1s, before the test rig's 0.5s poll interval caught the open port.

**The smoke-import check did NOT fire** because the start command was `python src/prime.py` (no `-m` flag). The check only activates on `-m` commands.

**Manifest:** `"start": "python src/prime.py"` ← wrong form.

### Critical Discovery: `python src/prime.py` Persists Despite Spec Fix

**All three generations produced `python src/prime.py` in their manifest `entry.start`.** This is the same root cause as gen-16, which prompted the spec v0.14.2 fix. The spec now says `python -m src.prime`, but the LLM consistently reverts to the direct-file form.

The smoke-import check (Phase 1) only fires when `-m` is present. Gen-18's failure was exactly the gen-16 pattern — but it bypassed the check entirely because no `-m` was used. The spec guidance was insufficient.

**Two follow-on actions needed before next run:**

1. **Manifest validation gate:** Add a validator in `test-rig/test_rig.py` that rejects any `entry.start` of the form `python <file.py>` when the file lives under a package directory (i.e., when `src/__init__.py` exists). This turns the failure into a manifest error at Stage 1 — immediate, with a clear error message — rather than an opaque crash at Stage 4.

2. **Spec reinforcement:** The spec's guidance on `entry.start` is still not emphatic enough. Even with the callout box added in v0.14.2, the LLM is not following it. The manifest-validation gate is the more reliable fix (mechanical enforcement vs. prose guidance).
