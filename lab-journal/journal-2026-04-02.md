## Phase 1: Smoke-Import Pre-Start Check (cambrian-tbnt)

**Motivation:** Gen-16 failed at the `start` stage with `ModuleNotFoundError: No module named 'src'`. The root cause was `python src/prime.py` vs `python -m src.prime` — a `sys.path` difference that tests can't catch because pytest adds CWD to `sys.path` automatically. The fix (`python -m src.prime`) was added to the spec in v0.14.2, but structural failures like this can still occur if the LLM mis-generates the import style. An early check costs <1s and surfaces the error before the process launch.

**Implementation:** Added `run_import_check(entry)` to `test-rig/test_rig.py`:
- Extracts the module name from `-m <module>` in `entry.start` using `shlex.split()`
- Runs `sys.executable -c "import <module>"` in WORKSPACE
- Returns `passed=True` immediately if no `-m` flag (fallback: start command doesn't use module form)
- Reports failure under the `start` stage — no schema change

The check runs as the first thing in Stage 4 (start), before `start_process()`. If it fails, the diagnostics include the stderr from the failed import (which is the actual `ModuleNotFoundError` message), making the root cause immediately visible in the viability report.

**Tests added:** `TestRunImportCheck` in `test-rig/test_test_rig.py` — 5 tests covering: no `-m` flag, no start command, importable module, missing module, `python3` variant.

## Phase 2: Parser Test Vectors in Spec (cambrian-jaxs)

**Motivation:** Gens 5 and 7 failed with identical bugs in `test_parse_multiple_close_tags_in_content`. Both generations implemented `"</file:end>" in line` (substring match) instead of `line.rstrip("\n\r") == "</file:end>"` (exact match). This is a systematic LLM blind spot: the spec described the state machine correctly but didn't call out the substring-vs-exact distinction, and provided no test vectors to catch the mistake.

**Changes to `spec/CAMBRIAN-SPEC-005.md`:**

Added two paragraphs after the state machine code block (existing line 338):
1. A callout explicitly naming the mistake: `"</file:end>" in line` vs `line.rstrip("\n\r") == "</file:end>"`, with an example showing `x = "</file:end>"` does NOT close the block.
2. Three test vectors as executable Python assertions:
   - Vector 1: `</file:end>` embedded in a longer line — not a close tag
   - Vector 2: Multiple files in sequence — each closed by their own `</file:end>`
   - Vector 3: `</file:end>` alone on its own line always closes the block; content after it is silently discarded (not a `ParseError`)

**Why these three:** Vector 1 pins the exact-match requirement (catches the `in line` bug). Vector 2 ensures the state machine resets correctly after each close (catches state-leak bugs). Vector 3 clarifies the "outside-any-block" behavior (discarded, not an error) — a common secondary misunderstanding.
