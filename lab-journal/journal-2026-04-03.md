## Critical Finding: Spec Guidance Is Not Effective for entry.start

**All 10 gens (20–29), and by extension all 14 gens since gen-16, fail for exactly the same reason.** The LLM consistently generates `python src/prime.py` in the manifest's `entry.start` field despite:
- Spec v0.14.2 adding an explicit callout box saying to use module form
- Spec v0.14.3 adding a manifest validation gate with an explicit error message
- BO iteration 1 mutating the "Implementation Requirements" section (spec text closer to LLM's generation prompt)

The spec guidance approach has produced zero improvement across 14 gens. The LLM has a strong prior for `python src/prime.py` that overrides written instructions.

**The only fix that will work is mechanical normalization**, not more spec text.

### Proposed Fix

**Auto-normalize `entry.start` in the test rig's manifest validation step**: when `entry.start` matches `python[3]? src/prime.py`, replace it with `python -m src.prime` and log a warning, instead of rejecting the manifest. Alternatively, normalize in `prime_runner.py` after artifact generation, before submitting to the test rig.

This is a one-line change that will immediately unblock the campaign and allow gens to advance past Stage 1. The LLM's behavior is preserved in logs; only the execution path changes.

Bead for this fix: **cambrian-hyuy** (P0).

## Phase 3: Root Cause Correction (cambrian-hyuy)

Post-run analysis revealed the journal's diagnosis was wrong. The fix bead was updated.

**Actual root cause:** `supervisor/prime_runner.py:146` hardcodes `"start": "python src/prime.py"` in `_write_manifest()`. The manifest is **not LLM-generated** — it is built programmatically by the Supervisor. The LLM never sees or produces the manifest. The spec guidance added in v0.14.2 and v0.14.3 was never relevant to this failure.

The test rig's manifest validation gate (cambrian-0p3w) was working correctly — it caught a real bug in trusted infrastructure code, not an LLM compliance failure.

**Fix:** One line in `supervisor/prime_runner.py:146`:
```python
# Before:
"start": "python src/prime.py",
# After:
"start": "python -m src.prime",
```

**Test suite:** 293 passed (unchanged). The manifest validation tests (`test_script_form_rejected_when_init_py_exists` etc.) now serve as regression tests against reintroducing this bug.

**What the bead cambrian-hyuy originally proposed** (auto-normalize in test rig) was the correct symptom identification but wrong fix layer. Updated to reflect the actual fix.

## Phase 4: Viability Improvements (cambrian-jpoq, cambrian-xmtq)

Analysis of all 29 gens showed infrastructure bugs caused 81.8% of failures. The adjusted LLM viability rate (excluding infra) is ~58%. Two improvements to push it higher:

### cambrian-jpoq (P0) — Enrich system prompt

The system prompt was 12 lines while the spec is 800+. Every known LLM failure mode had guidance in the spec body but NOT in the system prompt, where instruction-tuned models pay the most attention. Added 11 lines covering:

- **structlog API**: first positional arg is the event key; never pass `event=` as kwarg (gen-17 fix)
- **Local test imports**: every test function must import its own symbols (gen-19 fix)
- **aiohttp_client fixture**: do NOT use deprecated `AioHTTPTestCase` (recurring pattern)
- **Streaming LLM calls**: must use `.stream()`, not `.create()` (recurring pattern)
- **kebab-case JSON fields**: wire format uses hyphens, not underscores

### cambrian-xmtq (P1) — Feed Test Rig diagnostics to next generation

The `_build_prompt()` in `prime_runner.py` already had a well-designed `failed_context` mechanism — it adds a `# Previous Attempt Failed` section with full failed source code + diagnostics. But `campaign.py` never used it. Changes:

1. **`generate_artifact()`**: Added `failed_context` parameter (was only used internally for parse retries)
2. **`campaign.py` loop**: Constructs `failed_context` from viability report diagnostics when a gen fails, passes it to the next `generate_artifact()` call
3. **Deferred cleanup**: Non-viable artifact dirs are kept alive until the next gen's prompt has been built (so `_build_prompt()` can read the failed source files), then cleaned up in the `finally` block
4. **Final cleanup**: Last failed artifact is cleaned up after the campaign loop ends
