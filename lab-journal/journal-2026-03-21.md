## Implementation Language Discussion

### The Question

Which language should Prime (the organism) be implemented in? The user posed 10 criteria:

1. **LLM generation accuracy** — which languages do LLMs produce correct code in most reliably?
2. **GPU compute** — does Mojo's GPU story matter? (Concluded: no, Prime is I/O bound)
3. **Rust compiler feedback** — could compiler errors improve the retry loop?
4. **Python ecosystem** — familiarity, shared stack with Supervisor/Test Rig
5. **Rust performance** — relevant? (Concluded: no, Prime waits on LLM API calls)
6. **Elixir hot reload** — relevant? (Concluded: no, CAMBRIAN regenerates from scratch)
7. **TypeScript/OpenClaw momentum** — large community, strong LLM support
8. **Prototype pragmatism** — M1 needs to work first, optimize later
9. **No runtime compiler needed** — unlike Loom, CAMBRIAN doesn't eval code at runtime
10. **Benchmarks** — want data, not vibes

### Benchmark Data

Sources: TechPanda LeetCode benchmark (Oct 2025–Feb 2026), SWE-bench composition, Claude capability reports.

| Language | Claude (est.) | GPT-5-mini | Gemini 3 Flash | Gap vs Python |
|----------|--------------|------------|----------------|---------------|
| Python | ~93% | 93% | 84% | baseline |
| Java | ~92% | 94% | 93% | ~0 |
| TypeScript | ~90% | ~90% | ~85% | -3 to -5pp |
| Go | ~88% | ~88% | ~82% | -5pp |
| Rust | ~85% | 80% | 78% | -5 to -13pp |
| Elixir | ~70% | 63% | 83% | -15 to -43pp |
| Mojo | no data | no data | no data | unknown |

Key finding: LLM performance on coding problems correlates with language popularity (training data density). SWE-bench is almost entirely Python repos.

### Candidate Analysis

**Python** — Highest accuracy, shared stack, no compile step, Claude optimized for it. Downside: no static types by default, runtime errors surface late. Performance irrelevant (I/O bound).

**TypeScript** — 3-5pp behind, static types, Node.js runtime familiar from Loom. Downside: build step, already moved away from Node ecosystem.

**Go** — Non-obvious candidate. ~5pp behind Python. Simple language spec = fewer LLM failure modes. Single static binary = tiny Docker images. Fast compilation. Strong HTTP stdlib covers Prime's entire I/O surface.

**Rust** — 5-13pp behind. Excellent compiler errors for retry loops. But borrow checker is the #1 LLM stumbling block. Complexity creates more failure modes per generation.

**Elixir** — 15-43pp behind, largest gap, high variance. BEAM strengths (fault tolerance, concurrency) are overkill. Smallest training corpus.

**Mojo** — No benchmarks, immature ecosystem, GPU irrelevant. High risk for M1.

### Recommendation

**Python for M1. Revisit after reproduction works.**

Rationale:
1. Maximizes P(viable offspring) — the single metric for M1
2. Shared stack with Supervisor and Test Rig — one language, one Docker base image
3. No compile barrier — syntax errors produce useful messages immediately
4. Claude is most capable in Python (SWE-bench optimization, densest training data)
5. M1 is about reproduction, not performance — Prime is I/O bound

Secondary candidate: **Go** after Python M1 succeeds, for its simplicity and static binaries.

Key insight: once self-hosting works, the organism can be rewritten in any language — that's literally what CAMBRIAN does. A Gen-N Prime in Python could generate a Gen-N+1 Prime in Rust if the spec says so.

### Decision

**Python for M1.** User confirmed 2026-03-21. Spec updated (CAMBRIAN-SPEC-004, Implementation Language section, metadata `language: "python (M1)"`). The spec remains language-agnostic by design — future generations may use a different language.

## Python Version and Virtual Environments

**Decision:** Python 3.14 free-threaded build (`python3.14t`) everywhere. Virtual environments mandatory.

- Python 3.14 (released Oct 2025) has free-threading officially supported via PEP 779 — no longer experimental
- GIL is disabled, enabling true multithreading with ~5-10% single-threaded overhead
- Docker image: `python:3.14t-slim`
- Container venv: `/venv` created at image build time, activated via PATH
- Host venv: `.venv/` in project root for Supervisor and Test Rig
- Generated artifacts install into the container's `/venv`, never system Python

---

## Concurrency Model

**Decision:** `asyncio` everywhere for I/O-bound code. Combined with free-threaded Python 3.14t for true parallelism when needed.

- Supervisor: async HTTP server, non-blocking container polling via `aiodocker`
- Prime: concurrent LLM calls for parallel subsystem generation
- Test Rig: exception — stays sequential (linear pipeline, no benefit from async)

---

## Structured Logging

**Decision:** `structlog` for all components. JSON output in containers (machine-parseable), key-value on host (human-readable).

- Every log line: `timestamp`, `level`, `event`, `component`
- Generation-scoped operations include `generation` in log context

---

## Docker Container Management

**Decision:** `aiodocker` (async Docker SDK) instead of the sync `docker` SDK.

- Native `await` on container operations — no `run_in_executor` wrappers
- Fits the asyncio-everywhere principle
- Community-maintained, but our usage surface is small (create, wait, inspect, remove)
- Container stdout/stderr captured via `aiodocker` log streaming and attached to generation records for post-mortem debugging

---

## Type Safety

**Decision:** Full type annotations, strict enforcement in CI.

- **Type checker:** Pyright strict mode now, migrate to `ty` (Astral, Rust-based, 10-60x faster) when Pydantic plugin ships ([astral-sh/ty#2403](https://github.com/astral-sh/ty/issues/2403), currently 1/71 subtasks done)
- **I/O validation:** Pydantic v2 models are the single source of truth for all schemas in SPEC-004
- **Precise constructs:** `Protocol`, `TypedDict`, `Literal`, `Self` — `Any` prohibited
- **CI enforcement:** zero Pyright errors tolerated, build fails on type errors

---

## Runtime Introspection

**Decision:** Build everything for live debugging and introspection.

- `inspect` (stdlib) + `typing-inspect` for runtime type and call-stack introspection
- `devtools` for `debug()` calls during development
- `rich` for pretty-printing and `__rich_repr__` on all major objects
- Debug endpoints on Supervisor (`GET /debug/state`) for dumping internal state
- Generation failures log full stack + object state, not just error messages

---

## Serialization and JSON Handling

**Decision:** Pydantic v2 owns all JSON at I/O boundaries. No msgspec.

Evaluated msgspec (3-8x faster) but concluded it's unnecessary for our throughput profile:
- Prime is LLM-bound (seconds per call), not serialization-bound
- Supervisor handles one generation at a time
- Total JSON volume: ~10 documents per generation cycle (~1-2KB each)

Pydantic's strengths align directly with our goals:
- `model_validate_json()` for all deserialization at boundaries (type-safe, validated)
- `model_dump_json()` for all serialization at boundaries
- `model_json_schema()` for self-documenting APIs (pairs with introspection goal)
- `model_fields` for runtime introspection (pairs with `rich`/`devtools`)
- Rich validation errors (pairs with fail-loud principle)

Rule: raw `json.loads()`/`json.dumps()` MUST NOT be used at I/O boundaries — only for internal formatting where no validation is needed.

## Tooling Review

Reviewed the bootstrap spec for gaps given our use case (experimental, heavily-typed, async Python managing Docker containers and LLM calls). Identified seven additions:

### HTTP Server: `aiohttp`

**Decision:** Replace stdlib `http.server` + `urllib.request` with `aiohttp`.

- `http.server` is synchronous — wrapping it in asyncio is fighting the tool
- `aiohttp` provides native async HTTP server (Supervisor) and client (Prime → Supervisor API, health checks)
- Still no Flask/FastAPI — `aiohttp` is lower-level and fits our minimal-dependency ethos

### Project Management: `uv`

**Decision:** Use `uv` (Astral) for all package management.

- Replaces raw `pip install` and `python -m venv`
- `uv init`, `uv sync`, `uv add` for dependency management
- `uv.lock` committed to repo for reproducible builds
- 10-100x faster than pip
- Same Astral ecosystem as ruff (and eventually ty)

### Linting: `ruff`

**Decision:** `ruff` for linting and formatting. Zero errors in CI.

- Catches unused imports, dead code, style issues, security patterns
- Astral ecosystem, Rust-based, fast enough to run on every save
- Replaces flake8, isort, black in a single tool

### Git Operations: async subprocess

**Decision:** `asyncio.create_subprocess_exec("git", ...)` for all git operations.

- Keeps Supervisor non-blocking during promote/rollback
- No `gitpython` dependency — git CLI is universally available
- Operations are simple and infrequent (branch, merge, tag, delete)

### Async Testing: `pytest-asyncio`

**Decision:** Add `pytest-asyncio` for native async test support.

- All Supervisor and Prime code is async — need to test it natively
- `@pytest.mark.asyncio` on test functions, no manual event loop management

### Project Configuration: `pyproject.toml`

**Decision:** Single `pyproject.toml` for project metadata, dependencies, and tool configs.

- Standard for modern Python projects
- Contains ruff config, pyright config reference, uv metadata
- `pyrightconfig.json` alongside for Pyright strict mode settings

### Container Log Capture

**Decision:** Capture container stdout/stderr via `aiodocker` log streaming.

- Attached to generation records for post-mortem debugging
- When a generation fails without writing a viability report, logs are the only diagnostic
- Critical for debugging LLM output parsing failures inside containers

## Dolt Server Fix

**Root cause:** `bd` auto-starts a Dolt sql-server per command and stops it after. Unclean shutdowns leave zombie processes holding the database lock. Next command can't acquire the lock → "database is locked" error.

**Fix:** Updated SessionStart hook in `~/.claude/settings.json`:
```
bd dolt killall -q → bd dolt start -q → bd prime
```
Kills orphans, starts a persistent server for the session, then outputs workflow context. Server stays running instead of start/stop per command.
