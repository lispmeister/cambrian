## Phase 1: Security Fixes (cambrian-7gd0)

### 1a. API Key Removed from Test Rig Container Environment

**Problem:** `run_test_rig()` in `supervisor.py` was passing `ANTHROPIC_API_KEY` to the Docker container `env_list`. The Test Rig makes zero LLM calls — there is no legitimate reason for the key to be present. An adversarially-evolved Prime running during build, test, or start stages could read the key from its own environment.

**Fix:** Removed `api_key = os.environ.get("ANTHROPIC_API_KEY", "")` and `f"ANTHROPIC_API_KEY={api_key}"` from `run_test_rig()`. Same removal from the baseline reverse-run container.

### 1b. Network Isolation for Test Rig Containers

**Problem:** No `NetworkMode` was set — containers had full outbound network access. A compromised organism could exfiltrate data or call home during evaluation.

**Fix:** Added `"NetworkMode": "none"` to the HostConfig in `run_test_rig()` and the baseline reverse-run container. The health check uses `localhost:8401` which is intra-container loopback — confirmed safe with `NetworkMode: none`.

**New test:** `test_run_test_rig_no_api_key_in_env` asserts both that `ANTHROPIC_API_KEY` is absent from the env list AND that `NetworkMode` is `"none"`.

### 1c. Spec Updated (BOOTSTRAP-SPEC-002)

- §3.3 Credential Injection: completely rewritten to split Prime containers (key injected) from Test Rig containers (MUST NOT receive key); code example updated
- New §3.6 Network Isolation: Test Rig containers MUST use `NetworkMode: none`

## Phase 4: M2 Operational Readiness — R2 (cambrian-dm7t)

Three gaps that would prevent crash-safe M2 operation:

### 4a–4b. New Spec Sections (BOOTSTRAP-SPEC-002)

**§2.14 Adaptive Test Generation (M2):** Full spec for the adaptive test subsystem — purpose, trigger (failed campaign), LLM generation, storage schema (`adaptive-tests.json`), expiry mechanism, cap, and config table. This section was missing despite the feature being implemented.

**§2.15 Entanglement Monitoring (M2):** Design, computation (fraction of sections changed per mutation), threshold (mean ≥ 0.02 = entangling), BO loop integration.

### 4c. Generation Number Auto-Detection

**Problem:** `SpecBOLoop` hardcoded `start_generation=1`. Restarting M2 after an M1 run (or crash) would re-use generation numbers 1..N, corrupting the generation history.

**Fix:**
- `SpecBOLoop.__init__`: `start_generation: int | None = None`
- New `_detect_start_generation()`: async method that queries `GET /versions`, returns `max(generation) + 1` or 1 if empty
- Called in `run()` before the loop, with fallback
- `scripts/run_m2.py`: `CAMBRIAN_START_GENERATION` defaults to `"auto"`

### 4d. BO Observation Persistence + Crash Recovery

**Problem:** BO optimizer state (GP posterior) lived only in memory. A crash lost all evaluated points — the next run would start from scratch with no information about already-explored spec variants.

**Fix:**
- `_record_observation()`: appends each observation to `{artifacts_root}/bo-observations.jsonl` (spec_hash, spec_text, features, viability, summaries, target_section, timestamp)
- `_reload_observations()`: called in `__init__`; reads JSONL, reconstructs `BOObservation` objects, re-tells the optimizer. Graceful on corrupt file (logs warning, starts fresh).
- Generation counter derived from reloaded state (sum of mini + full campaign sizes per observation)

### 4e. Best Spec Metadata

`scripts/run_m2.py` now writes `best-spec-meta.json` alongside `best-spec.md`:

```json
{
  "spec_hash": "sha256:...",
  "viability_rate": 0.85,
  "iterations": 20,
  "budget_used": 135,
  "timestamp": "2026-04-09T..."
}
```

### 4f. Campaign-ID Filter on GET /versions

**Problem:** The campaign polling loop in `run_campaign()` fetched ALL generation records on every poll cycle. As the generation history grows (M1 + many M2 campaigns), this becomes unnecessarily expensive.

**Fix:**
- `handle_versions()`: accepts optional `?campaign-id=X` query parameter; filters when present
- `campaign.py` polling loop: constructs `versions_url = f"{supervisor_url}/versions?campaign-id={campaign_id}"`

### 4g. CAMBRIAN-SPEC-005 M2 Deferred Callouts

Added explicit `> **Implementation status:** ... deferred to Stage 2+` callouts to:
- MAP-Elites section
- Meta-Monitor section

Added "Stage 1 persistence" paragraph documenting `bo-observations.jsonl` as the crash recovery mechanism.
