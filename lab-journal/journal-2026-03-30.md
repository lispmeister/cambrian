## Anti-Cheating Verification Layers

Identified the core problem: a generation controls its own tests, so it can "cheat" by writing trivial tests that pass without verifying the spec. Designed and specced three verification layers:

### Layer 1 — Spec Acceptance Vectors (M1, mechanical)

5 HTTP test vectors in a FROZEN block in CAMBRIAN-SPEC-005. The Test Rig reads them directly from the spec file in `/workspace` and evaluates them during the health stage, before manifest contracts. The offspring cannot modify FROZEN blocks.

Vectors: `sv-health-liveness`, `sv-health-body`, `sv-stats-schema`, `sv-stats-generation`, `sv-stats-status-is-string`.

### Layer 2 — Dual-Blind Examiner (M2 Tier 1, LLM)

An independent LLM generates test cases from the spec alone, without seeing the offspring's code. The Supervisor orchestrates this after the Test Rig completes. Code author and test author share only the spec — no channel to collude.

### Layer 3 — Adversarial Red-Team (M2 Tier 2, LLM)

An independent LLM reads the spec AND the offspring's source code. Goal: find violations. Produces failing test cases. If any test reveals a real bug, the offspring's fitness is penalized.

| Layer | Trust | LLM? | Activation |
|-------|-------|------|------------|
| 0: Self-Tests | Offspring controls | No | Always |
| 1: Spec Vectors | FROZEN in spec | No | Always (M1) |
| 2: Dual-Blind | Independent LLM | No collude | M2 Tier 1 |
| 3: Red-Team | Independent LLM | Hostile | M2 Tier 2 |
