## Bootstrap Spec Evolution: BOOTSTRAP-SPEC-001 → BOOTSTRAP-SPEC-002

Produced BOOTSTRAP-SPEC-002 as a derivative of BOOTSTRAP-SPEC-001. Four major additions, each building on the last. The session was structured as a series of "what's the single most impactful addition?" prompts — forcing prioritization over breadth.

### Addition 1: Verification Contracts (§2.5)

**Problem:** The Test Rig's health-check stage hard-codes two checks (`GET /health` → 200, `GET /stats` → JSON with `generation`). In M2, when specs mutate, organisms may expose different endpoints. Every spec mutation would require a Test Rig code change.

**Solution:** A `contracts` array in the manifest — declarative HTTP checks that the organism declares and the Test Rig evaluates. Three match modes: `body` (exact), `body_contains` (partial), `body_has_keys` (structural). Variable substitution (`$GENERATION`) for dynamic values. Backward compatible — absent contracts fall back to hard-coded checks.

**Key insight:** The organism defines its own fitness criteria; the environment enforces them. This decouples organism evolution from environment code.

**Bead:** cambrian-05h

### Addition 2: Structured Diagnostics (§2.6)

**Problem:** Loom's 1.4% viability rate (1/72) was partly because the LLM had no structured feedback about WHY previous attempts failed. The viability report reduced everything to a boolean.

**Solution:** A `diagnostics` object in the viability report when `status` is `non-viable`. Contains: `stage`, `summary` (one-line, LLM-optimized), `exit_code`, `failures[]` (per-test with name/error/file/line), `stdout_tail` (last 100 lines), `stderr_tail` (last 100 lines). Per-stage formats defined for all five pipeline stages. Pytest failure extraction via `FAILED <file>::<test> - <error>` pattern.

**Key insight:** This turns the Test Rig from a Darwinian oracle (binary pass/fail, blind retry) into a Lamarckian teaching signal (structured failure context feeds into the next generation's prompt). The organism doesn't mutate randomly — it learns what went wrong.

**Bead:** cambrian-2hx

### Addition 3: Informed Retry (§2.7)

**Problem:** Diagnostics tell Prime WHAT failed, but Prime never sees the code that caused the failure. After rollback, `git branch -D gen-N` deletes the failed code. The LLM retries from scratch with no knowledge of its previous attempt.

**Solution:** Rollback now tags the failed branch (`gen-N-failed`) before deleting it. Generation records gain an `artifact_ref` field pointing to the tag. Prime reads its own previous failed code via `git show gen-N-failed:<file>` and includes it with diagnostics in the retry prompt. The LLM goes from "try again" to "here's what you wrote, here's what broke, fix it."

**Key insight:** Retries within `CAMBRIAN_MAX_RETRIES` are Lamarckian (informed by failure). Generations across spec versions (M2) are Darwinian (clean-room). Two evolutionary mechanisms at different timescales. Each retry still produces a COMPLETE artifact — no patching, no path dependence.

**Bead:** cambrian-hfh

### Addition 4: Fitness Vector (§2.8)

**Problem:** Viability is binary. All viable organisms are treated as equally good. When M2 arrives and you need to select between organisms, there's no historical data.

**Solution:** A `fitness` object in every viability report — 15 quantitative metrics across four dimensions: Speed (build/test/start/health durations), Correctness (test count, pass rate), Economy (token usage, source lines, dependency count), Robustness (test files, test lines, test-to-source ratio). All computed from data the Test Rig already collects. Zero additional test execution.

**Key insight:** The measurement apparatus is defined now; the selection policy is deferred to M2. Historical fitness data exists from Gen-1 onward — no retrofitting required.

**Bead:** cambrian-fy3

## Architectural Insight: The Feedback Stack

The four additions form a coherent stack that transforms the retry loop:

```
§2.5 Contracts    — organism declares what it promises
§2.6 Diagnostics  — environment explains why it failed
§2.7 Informed Retry — organism reads its own failed code
§2.8 Fitness      — environment measures how good it is
```

Without this stack: spec → LLM → code → pass/fail → blind retry (Loom: 1.4% viability)
With this stack: spec → LLM → code → structured failure + source + metrics → informed retry
