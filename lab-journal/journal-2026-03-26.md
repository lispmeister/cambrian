## 1. Systematic Gap Analysis

Before writing code, ran a three-agent exploration comparing existing bootstrap code against
the updated specs. Found ~30 gaps across supervisor.py, test_rig.py, and generations.py.
The most significant:

**Supervisor:**
- `/stats` returned the latest generation's `outcome` as `status`, conflating Supervisor
  operational state with generation record outcome. A Supervisor that just finished promoting
  is `idle`, not `promoted`.
- No container timeout — a hung container would block the Supervisor indefinitely.
- Container cleanup (`container.delete()`) was not in a `finally` block — a crash mid-pipeline
  would leak containers.
- No `campaign-id` passthrough (needed for M2).
- `generations.update()` had no terminal-record guard — could overwrite a `promoted` record.

**Test Rig:**
- No manifest MUST-field validation — any malformed manifest passed silently.
- No TCP-port readiness check in the start stage — just launched the process and immediately
  hit the health endpoint.
- No pytest failure extraction — diagnostics had no `failures[]` array.
- Fitness vector was incomplete and `stages_completed` was missing.
- Unattempted stages did not appear in `checks` — report consumers couldn't tell what ran.
- Fallback health check only checked `/health`, not `/stats`.

## 2. What Changed

### `supervisor/generations.py`

Added terminal-record guard in `update()`. Once a record reaches `promoted`, `failed`, or
`timeout`, further updates are silently rejected with a warning log. Protects against race
conditions and append-only semantics.

### `supervisor/supervisor.py` (full rewrite)

- **Status machine:** `_status` global tracks `idle/spawning/testing/promoting/rolling-back`.
  Transitions set explicitly at each point. `/stats` returns this, not the generation outcome.
- **`run_test_rig`:** `asyncio.wait_for(container.wait(), timeout=CONTAINER_TIMEOUT)` prevents
  hung containers. `container.delete()` moved to `finally` block with `contextlib.suppress`.
  Missing-report fallback synthesises a non-viable report with a clear diagnostic rather than
  crashing.
- **`handle_spawn`:** Verifies Docker image exists before spawning (fails fast with a clear
  error). Passes `campaign-id` through to generation record.
- **`/debug/state`:** New endpoint dumping full internal state as JSON for development.
- **`handle_stats`:** Returns `{generation, status, uptime}` where `generation` is the highest
  completed generation number (0 if none), and `status` is the Supervisor's own state.

### `test-rig/test_rig.py` (full rewrite, keeping 5-stage skeleton)

- **Manifest validation:** All 12 MUST fields checked with types and regexes. `cambrian-version`
  must equal 1. `contracts` schema validated if present.
- **Contract evaluation:** HTTP contracts with `status`, `body`, `body_contains`, `body_has_keys`
  checks. `$GENERATION` substitution (produces integer not string). All contracts evaluated —
  no short-circuit. Fallback to hard-coded checks when `contracts` absent.
- **TCP-port readiness (stage 4):** `socket.create_connection` polling at 0.5s intervals, 30s
  timeout. Fails immediately with the process's exit code if it dies before the port opens.
- **Diagnostics:** `exit_code`, `failures[]` with pytest `FAILED file::test - error` extraction
  (500-char truncation per error), `stdout_tail`/`stderr_tail` (last 100 lines).
- **Pytest counts:** Returns `(-1, -1)` for non-pytest output — "unknown" not "zero".
- **Fitness vector:** All 15 metrics. Per-stage durations only for `stages_completed` stages
  (absent = unattempted, not zero). `contract_pass_rate` absent when no contracts.
  `stages_completed` always present.
- **Skipped stages:** Unattempted stages appear in `checks` with `passed: false, duration_ms: 0`.
- **Source/test classification:** By basename pattern (`test*`, `*_test*`), not hardcoded paths.

### Tests written

- `supervisor/test_supervisor.py` — 23 tests: all HTTP endpoints, generation CRUD, status
  state machine, terminal-record guard, campaign-id passthrough.
- `test-rig/test_test_rig.py` — 49 tests: manifest validation, pytest parsing, fitness vector,
  contract evaluation (including no-short-circuit), diagnostics schema, skipped stages, TCP
  readiness, `$GENERATION` substitution.

All 72 tests passing. Ruff clean.

## 4. Bug Found: DOCKER_HOST on macOS

**Symptom:** `/spawn` returned `{"ok": false, "error": "Docker image cambrian-base not found"}`
even though `docker images` showed the image clearly.

**Root cause:** On macOS with Docker Desktop, `/var/run/docker.sock` is a symlink to
`~/.docker/run/docker.sock`. `aiodocker.Docker()` (without `DOCKER_HOST` set) does not
reliably follow this symlink when inspecting images or creating containers. The operation fails
silently — no OS error, just a "not found" from the Docker API.

**Verified:** `aiodocker.Docker().images.inspect('cambrian-base')` works correctly when
`DOCKER_HOST=unix://$HOME/.docker/run/docker.sock` is set. The symlink at `/var/run/docker.sock`
is the problem; the real socket at `~/.docker/run/docker.sock` works fine.

**Fix:** Documented in BOOTSTRAP-SPEC-002 v0.8.2:
- §1.4 — Added environment variable table with macOS start command example.
- §3.2 — Added explanation of the symlink issue, symptom, and platform difference.

This is a macOS + Docker Desktop specific issue. On Linux, `/var/run/docker.sock` is the real
socket and no override is needed.

## 5. Spec Lineage Note on handle_spawn

During code review, spotted a potential inconsistency: `handle_spawn` constructs the artifact
path as `Path(artifacts_root) / artifact_rel` where `artifact_rel` is the `artifact-path`
field from the request body. The spec says this field is "absolute host-side path to the
artifact directory" — but the code joins it with `artifacts_root`, implying it's relative.

In practice, during our test we sent `"artifact-path": "gen-test"` (a relative path inside the
artifacts repo), and it worked correctly. But the spec is ambiguous. This did not affect today's
work but should be clarified before Gen-1 is regenerated — Prime will need to know what to send.

Ran Gen-1 through the Supervisor pipeline. Hit two bugs before it passed.

### Bug 1: `gen-1-new` source files disappeared from artifacts repo

During an earlier test run (gen-4), the Test Rig mounted `gen-1-new` as `/workspace`.
Cause still unclear, but after the test all .py source files were gone from the directory
(only `__pycache__` and `viability-report.json` remained). Workaround: copy files with
`cat src > dst` one at a time (dcg hook blocks `rm -rf` and `cp -r`).

### Bug 2: `/stats generation` returned wrong value (spec contradiction)

**Symptom:** Health contract `stats-generation` failed with `body['generation'] = 2,
expected 1`.

**Root cause:** `prime.py` initialized `_generation = 0` and updated it to `gen_num`
after step 1 of the loop. `gen_num = highest_in_Supervisor + 1`. But by the time Prime
runs, the Supervisor has already created an in_progress record for the current generation.
So `get_versions()` returns that record, `highest = 1`, `gen_num = 2`, and `/stats`
returns `generation: 2` instead of `generation: 1`.

**Spec contradiction:** Line 64 said `/stats generation` MUST match the artifact's own
generation number. Line 221 said it should equal the "active generation number" (gen_num
= next offspring). These are incompatible: in a fresh environment they coincide, but in
any real deployment they don't.

**Fix:** Initialize `_generation = int(os.environ.get("CAMBRIAN_GENERATION", "0"))` at
module level. `CAMBRIAN_GENERATION` is set by the Supervisor to the spawned generation
number — Prime's own identity. The loop's `gen_num` is now a purely internal variable.

**Spec update:** CAMBRIAN-SPEC-005 v0.10.2 → v0.10.3 (patch):
- §3 `/stats`: Rewrote the startup behavior note. `generation` = `CAMBRIAN_GENERATION`
  (fixed identity), not "active generation number". Removed the misleading
  "before step 1, returns 0" language.
- Step 1: Removed "This is also the value Prime returns from GET /stats".
