## Research: A-Evolve and Adversarial Review

### A-Evolve (Amazon / Henry Lu)

Added `research/07-agentic-evolution-infrastructure.md`. Key findings:

- A-Evolve is a file-system-based agent evolution framework. Agents work in isolated workspaces; the runtime observes performance metrics, generates mutations, and applies them to the next generation.
- Self-describes as "PyTorch for Agentic AI" — infrastructure layer, not a specific evolved system.
- The linked arXiv paper (2602.00359) proposes "evolution compute" as a third scaling axis alongside pre-training and inference compute. The evolution-scaling hypothesis: adaptation quality grows with evolution compute.
- A-Evolve's 5-phase loop (Solve → Observe → Evolve → Gate → Reload) maps closely to Cambrian's own loop, validating the general architecture.
- Key gap: A-Evolve evolves agent *behavior* (tool use patterns, prompting strategy), not the agent's *source code + spec*. Cambrian evolves the full genotype including self-description.

**Cambrian uniqueness narrowed but real:** The original claim that "no system evolves structured specifications" is too strong — PromptBreeder, DSPy, TextGrad all evolve structured text. Cambrian's actual novelty is the *genotype-phenotype indirection*: the organism generates its own replacement source code from a natural-language spec, plus *recursive self-reproduction* (offspring can reproduce again) and *document-scale genomes* (spec is too long for prompt optimization baselines).

### Adversarial Review (Round 2)

Three parallel adversarial agents challenged every round-1 assumption. Added `research/08-adversarial-review.md`. 10 findings:

1. **Container isolation gap** — organism and test rig share a filesystem volume. Under selection pressure, a spec could evolve code that writes a fake viability report. Must be addressed before M2 population runs. → bead cambrian-2cf (P1)
2. **MAP-Elites wrong for budget** — needs 200+ evaluations for meaningful coverage. With 20-50 attempts per campaign, use BO→BOP-Elites staged approach.
3. **Dual-model screener hackable** — LLM-based screening of generated code can itself be gamed by evolved code. Mini-campaign screening (run it, measure it) is more robust.
4. **Monolithic spec → entanglement** — a long, dense spec creates high coupling between evolved features. Added modularity pressure + entanglement monitoring to M2 proposal.
5. **Grammar-constrained mutation superior** — unconstrained LLM mutation of specs wastes attempts on syntactically invalid variants. Formal spec grammar would gate mutations. → bead cambrian-2e5
6. **Self-referential fitness discounting** — if the fitness function is itself part of the evolving spec, it will evolve to trivially satisfy itself. Added discounting mechanism to proposal.
7. **Adaptive test generation (Tier X)** — purely static tests get saturated; LLM-generated tests should chase evolved behavior. Added to proposal.
8. **Temperature annealing** — fixed neutral-drift temperature wastes budget. Anneal from high (exploration) to low (exploitation).
9. **DGX motivation overstated** — prior art on spec evolution existed; thesis was revised.
10. **Neutral drift waste** — the proposal relied on random drift for diversity. Replaced with directed diversity pressure (BOP-Elites).

`research/06-proposal.md` updated v0.1 → v0.2 incorporating all 10 findings.

## Spec Patches (M1 Blockers)

Two spec beads were root-cause for gen-2 test failures:

### cambrian-ux4: Streaming requirement

Gen-2 used `client.messages.create()` with `max_tokens=32768`. The Anthropic SDK raises a timeout error for large `max_tokens` with non-streaming calls. Fix: added explicit streaming rule to CAMBRIAN-SPEC-005 `§Language and runtime` and to the gen-1 `SYSTEM_PROMPT`:

```
MUST use `async with client.messages.stream(...) as stream:` with
`await stream.get_final_message()`. Do NOT use `client.messages.create()`.
```

### cambrian-5kb: aiohttp test pattern

Gen-2 used `AioHTTPTestCase.get_client(self.server)` — removed in aiohttp 3.8+. Added a concrete `aiohttp_client`/`aiohttp_server` fixture example to CAMBRIAN-SPEC-005 `§Tests` and to `SYSTEM_PROMPT`. Also specified `asyncio_mode = "auto"` in pytest.ini.

Both patches committed to CAMBRIAN-SPEC-005 (v0.11.0) and to the gen-1 artifact's `SYSTEM_PROMPT`. Spec hash in `manifest.json` updated accordingly.

### What the failures reveal

- **Gen-2**: Both spec patches were needed. After patching, no further streaming or aiohttp errors.
- **Gen-3, 5, 7**: Test-level failures from LLM-generated tests asserting incorrect behavior. Not spec bugs — the spec is silent on edge cases that tests exercised. Each retry self-corrected.
- **Pattern**: 1 spec issue (generations 2-3), then clean sailing from gen-4 onward except occasional test-level mismatches that auto-corrected.

### Gen-5 edge case worth noting

`test_parse_multiple_close_tags_in_content` expected the state machine to stop at an *inline* `</file:end>` within a line. The spec-correct behavior (and the gen-1 implementation) only stops at a line that *is* `</file:end>`. The test was wrong, not the implementation. This was self-corrected on retry but recurred. Consider adding a clarifying example to the spec.
