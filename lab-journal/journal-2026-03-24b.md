## Blocking Issues Fixed

### 1. BOOTSTRAP-SPEC-001 vs BOOTSTRAP-SPEC-002

**Problem:** Both specs existed with no deprecation notice. CLAUDE.md referenced SPEC-001. The two specs had conflicting rollback behavior (SPEC-001 deletes branches without preserving failed artifacts; SPEC-002 creates `gen-N-failed` tags).

**Fix:** Added a deprecation notice to BOOTSTRAP-SPEC-001 with a summary of the differences. Updated CLAUDE.md project structure and references section to point to BOOTSTRAP-SPEC-002.

### 2. Test artifact violates its own manifest schema

**Problem:** The test artifact used `"generation": 0` but SPEC-004 required `>= 1`. Also, the test artifact's `files` array omitted the spec file, which SPEC-004 required.

**Fix:** Updated the `generation` field rule in SPEC-004 to allow `>= 0`. Generation 0 is now explicitly defined as the valid value for hand-crafted test artifacts only; LLM-generated artifacts use `>= 1`. The test artifact is hand-crafted infrastructure, not an LLM output — different rules apply.

The `files` array issue is resolved by context: the test artifact's `spec-hash` is zeroed out (`sha256:000...`) and `producer-model` is `"hand-crafted"`, clearly marking it as a special case. No spec file is needed in the test artifact's `files` array because there is no spec to include.

### 3. `generations.json` "append-only" contradicted update requirement

**Problem:** The spec said records MUST NOT be modified, but promote/rollback must update `outcome` from `in-progress` to terminal.

**Fix:** Clarified the semantics in both SPEC-004 and BOOTSTRAP-SPEC-002: "append-only" means no record is deleted and no *completed* record is modified. The in-progress → terminal transition is a lifecycle update, not a correction, and happens exactly once. This is the standard meaning of "append-only audit log" in practice.

### 4. `CAMBRIAN_SPEC_PATH` defaulted to the wrong file

**Problem:** SPEC-004's configuration table defaulted `CAMBRIAN_SPEC_PATH` to `./spec/CAMBRIAN-SPEC-004.md`. But Prime should read the genome (SPEC-005), not the system spec.

**Fix:** Corrected the default to `./spec/CAMBRIAN-SPEC-005.md`. Also corrected `CAMBRIAN_MODEL` default from `*best available*` to the concrete value `claude-opus-4-6`.

### 5. Git from inside containers — design decision

**Problem:** The examples showed Prime doing `git checkout -b gen-1 && git add -A && git commit`. But Prime runs inside a Docker container with only `/workspace` bind-mounted. The host's git repository is inaccessible from inside the container.

**Design decision: Supervisor owns all git operations.**

Rationale: This is consistent with "Supervisor manages infrastructure." The Supervisor is the host-side process with access to the git repository. Giving Prime git access would require mounting the full host repo into the container — a security risk and an architectural violation (infrastructure leaking into the organism).

**New flow:**
1. Prime writes generated files to `/workspace/gen-N/` (a subdirectory of the bind-mounted workspace)
2. Prime computes hashes, writes `manifest.json`
3. Prime calls `POST /spawn` with `artifact-path: /workspace/gen-N`
4. **Supervisor** creates the `gen-N` git branch, copies/commits artifact files, then spawns the Test Rig container
5. Test Rig runs; Supervisor updates generation record when done
6. Prime polls `GET /versions` until terminal outcome appears

**New configuration variable: `CAMBRIAN_WORKSPACE_ROOT`** — the host-side directory bind-mounted as `/workspace` inside containers. Default: `./workspace`. The Supervisor resolves relative `artifact-path` values against this, and Docker bind mounts require absolute host paths.

**Spec changes:** Updated SPEC-004 examples (Example 1, Example 2), spawn endpoint notes, and configuration table. Prime's step 7 now says "Prime MUST NOT perform git operations."

## High-Priority Ambiguities Fixed

### 6. `artifact-path` — absolute vs relative

Added to SPEC-004 spawn notes: `artifact-path` must be absolute, or relative to `CAMBRIAN_WORKSPACE_ROOT`. The Supervisor resolves it before performing Docker bind mounts.

### 7. Spawn async semantics

**Problem:** The spec said `/spawn` returns immediately, but the code example showed a synchronous flow.

**Fix:** Updated BOOTSTRAP-SPEC-002 to show the correct async pattern. The `spawn_handler` performs the git operations (synchronous setup), creates the container, returns immediately, and then `asyncio.create_task(run_test_rig(...))` runs the Test Rig as a background task. The generation record is updated by the background task when the Test Rig completes. The code example now correctly reflects this split.

### 8. `$GENERATION` type coercion

**Problem:** `{"generation": "$GENERATION"}` — after string substitution, would `$GENERATION` become string `"0"` or integer `0`? The actual stats response has integer `0`.

**Fix:** Added a clarification note: when `$GENERATION` appears as a JSON value (not inside a larger string), it is replaced with the integer value, not a string. The Test Rig implementation must handle this by parsing the substitution context and emitting the correct JSON type.

### 9. Supervisor needs `artifact-hash` — where does it come from?

**Problem:** Generation records require `artifact-hash` but the spawn request doesn't include it.

**Fix:** Specified in SPEC-004 spawn notes: the Supervisor reads `artifact-hash` from `manifest.json` in the artifact directory. The Supervisor trusts the manifest's self-reported hash in M1 (Prime computes it). Independent verification of artifact integrity is an M2+ concern.

### 10. Gen-1 autonomous start — no API for it

**Problem:** The Supervisor's `/spawn` creates Test Rig containers, not Prime containers. There was no documented way to start Gen-1 Prime for autonomous reproduction.

**Fix:** Added a `docker run` command example to Stage 2 of BOOTSTRAP-SPEC-002 with the required flags (workspace mount, spec mount read-only, env vars, host networking for Supervisor access). Noted that a `/start` endpoint may be added in M2+ but is out of scope for M1 (which is explicitly a human-supervised bootstrap).
