## 1. Infrastructure Bugs Found and Fixed

### 1.1 Streaming requirement (cambrian-svf)

Gen-1's `generate.py` used `client.messages.create()` with `max_tokens=32768`. The Anthropic
SDK rejected this with:

```
Streaming is required for operations that may take longer than 10 minutes.
```

**Fix:** Changed `call_llm()` to use `async with client.messages.stream() as stream:` with
`await stream.get_final_message()`. This is the correct pattern for large outputs; the spec
already stated "Streaming MUST be used" (BOOTSTRAP-SPEC-002 §… the system prompt note) but
the gen-1 implementation missed it. Fixed in `gen-1-new/src/generate.py` and manifest hash
updated.

### 1.2 Git state bug: `generations.json` branch conflict

The Supervisor's spawn handler creates a branch (`gen-N`), commits the artifact with
`git add -A`, and stays on that branch. When the test rig finishes and the next spawn is
called, `ensure_on_main()` runs `git checkout main` — which fails:

```
error: Your local changes to the following files would be overwritten by checkout:
    generations.json
```

Root cause: `generations.json` was committed on the `gen-N` branch (via `git add -A`), then
updated by the Supervisor after the test run. Checking out `main` would overwrite those
uncommitted updates.

**Fix:** Added `generations.json` to `.gitignore` in the artifacts repo and ran
`git rm --cached generations.json` to untrack it. Now:

- `git add -A` never stages `generations.json` (ignored)
- Branch switches never conflict on it — it persists in the working tree untouched
- The file accumulates generation history without git tracking, managed entirely by the
  Supervisor process

This also means `git branch -D gen-N` in the rollback path now works (we return to main
cleanly before the next spawn).

BOOTSTRAP-SPEC-002 does not document this requirement; a follow-up patch will add it.

## 3. Root Cause: Python 3.14 String Literal Restrictions

The LLM consistently generated test code like this:

```python
close_tag = "
</file>"
```

Python 3.14 (following the deprecation in 3.12) treats an unescaped newline inside a
regular string literal as a `SyntaxError: unterminated string literal`. The correct form
is either:

```python
close_tag = "\n</file>"      # escaped newline
close_tag = """
</file>"""                   # triple-quoted
```

This failure appeared in `tests/test_generate.py`, which tests the `parse_files()` function.
That function extracts `<file path="...">content</file>` blocks from LLM responses. Writing
test strings for this naturally involves multi-line file content, which is where the LLM
went wrong.

The retry prompt included the full error message and failed source code, but Sonnet 4.6
repeated the same mistake across three separate retries. The model either lacks training
data on Python 3.14's strictness or doesn't connect the error message to the fix.

---

## 3a. Deep Analysis: What the LLM Is Trying to Quote

All failures in `test_generate.py` trace to the same functional need: the file contains tests
for `parse_files()`, which extracts `<file path="...">content</file>` blocks from raw LLM
response strings. To test this function, the test code must construct Python string literals
that *look like LLM responses* — i.e., strings containing `<file>` tags with multi-line content
inside them.

This creates a three-level quoting structure:

```
Python code → strings representing LLM output → strings representing Python code
```

The gen-1-new reference implementation (written by a human) avoids the problem with string
concatenation and `\n` escapes:

```python
response = (
    '<file path="src/a.py">a = 1</file>\n'
    '<file path="src/b.py">b = 2</file>'
)
```

The LLM, generating offspring, reaches for the "obvious" form — a string with a real embedded
newline — which was legal in Python ≤3.11 and is now a hard error in 3.14.

The failure modes across generations were:

| Gen | Location | Pattern | Error |
|-----|----------|---------|-------|
| 2, 4 | `test_generate.py:29` | `close_tag = "` + actual newline + `</file>"` | unterminated string literal |
| 5 | `generate.py:24` | SYSTEM_PROMPT written as single-quoted string with embedded newlines | unterminated string literal |
| 6 | build | `requirements.txt` absent — parse failure, incomplete output | build failure |
| 7 | `test_generate.py:39` | `"""Return:  ` with trailing whitespace after `"""` | unterminated triple-quoted literal |

Gen-5 is the most instructive: the LLM failed to apply the Python 3.14 rule to the
**SYSTEM_PROMPT constant itself** in `generate.py`. The SYSTEM_PROMPT is a multi-line
instruction string; the LLM wrote it as a single-quoted string with real newlines — the
exact pattern it was supposed to avoid. This means the problem isn't only about `<file>`
blocks in tests; it's a general failure to maintain the rule under the cognitive load of
generating a large codebase.

Gen-7 shows that retry feedback reached the LLM: it attempted triple quotes. But it
introduced trailing whitespace after the opening `"""`, which prevents Python from matching
the closing delimiter. The LLM understood the rule, applied it, and still got the syntax wrong.

**The structural issue:** the LLM's training data is overwhelmingly pre-3.12 Python, where
implicit line continuation inside string literals was legal (SyntaxWarning in 3.12, SyntaxError
in 3.14). The model has weak signal for this constraint. It can parrot the rule when prompted
but doesn't apply it reliably while generating hundreds of lines of code.

**Design question:** Could we avoid this class of failure by changing how test fixtures are
structured? Options:

1. **Spec provides a concrete fixture pattern** — give the LLM a copy-pasteable template rather
   than a rule to apply, e.g., "use this exact form for multi-line test strings in test_generate.py"
2. **Load fixtures from files** — store sample LLM response strings as `.txt` files in
   `tests/fixtures/`; no Python quoting issue
3. **Accept retry-convergence** — the rule is correct, question is whether 3 retries with
   Sonnet 4.6 is ever enough, or if escalation to a stronger model after N failures is needed

Option 2 trades a quoting problem for a fixture management problem (keeping the files in sync
with the parser spec). Option 3 is the minimal change — we already have the rule; we just need
the retry budget or escalation policy to match the failure rate.

## 4. Genome Gaps Identified

### 4.1 System prompt underspecifies Python 3.14

The SYSTEM_PROMPT says:

```
- The code must work in Python 3.14 inside a Docker container with a venv at /venv.
```

This states the requirement but not the constraints. "Works in Python 3.14" doesn't tell the
LLM what Python 3.14 enforces more strictly than 3.11 or 3.12. The spec needs to encode the
environmental fitness constraints precisely enough that the LLM can implement them correctly.

### 4.2 System prompt / spec divergence risk

The SYSTEM_PROMPT in `generate.py` is a hard-coded string, but it also appears verbatim in
CAMBRIAN-SPEC-005. These can diverge across generations. If the spec is updated but
offspring inherit the old system prompt from the parent's source code, the fix won't
propagate. For now both must be kept in sync manually; a future improvement is to have the
spec be the sole source of truth.

### 4.3 No retry escalation policy

After 3 identical failures, the system did nothing different. The spec doesn't define when
to escalate (use a stronger model, change the prompt strategy, or give up). This is an M2
concern — the organism should eventually be able to observe its own failure patterns and
mutate its genome to avoid them.

## 3b. Probe: Can Opus 4.6 Solve the Quoting Problem?

A focused probe script (`scripts/probe_opus_quoting.py`) was written to test the hypothesis
that the quoting failure is a model-capability issue solvable by escalation to Opus 4.6.

**Setup:** Send Opus 4.6 the exact SYSTEM_PROMPT from `gen-1-new/src/generate.py` (v0.10.4)
plus a focused task: "generate `tests/test_generate.py` with tests for `parse_files()`."
Validate the extracted file with `ast.parse()`. Run 5 attempts total.

**Results:**

| Attempt | Outcome |
|---------|---------|
| 1 | PASS (168 lines, syntax OK) |
| 2 | PASS (175 lines, syntax OK) |
| 3 | API error (Anthropic internal server error — not a model failure) |
| 4 | PASS (173 lines, syntax OK) |
| 5 | PASS (159 lines, syntax OK) |

4/4 generation attempts that completed produced syntactically valid Python 3.14. The probe
confirmed that Opus 4.6 correctly uses triple-quoted strings or `\n`-escaped strings when
constructing `<file>` block fixtures — exactly the pattern Sonnet 4.6 failed to apply.

**Conclusion:** The quoting failure is a model-capability issue. Escalating to Opus 4.6 is a
viable fix. This supports adding a retry-escalation policy: if Sonnet 4.6 fails N times on
the same generation, retry once with Opus 4.6.

**Second finding — parse_files() nested-tag bug:**

The initial probe used the same `FILE_PATTERN` regex as `parse_files()`:

```python
re.compile(r'<file path="([^"]+)">(.*?)</file>', re.DOTALL)
```

This non-greedy regex stops at the *first* `</file>` it encounters. When the LLM generates
`tests/test_generate.py`, the test fixture strings contain inner `<file path="...">...</file>`
patterns. The parser extracts only the fragment up to the inner `</file>` — a truncated,
invalid Python file. This means even a correctly-generated `test_generate.py` would be
corrupted by Prime's own extraction step.

This is an independent bug from the quoting issue. Even if Opus 4.6 generates perfect code,
the current `parse_files()` cannot correctly extract it. A fix to `parse_files()` is needed
alongside model escalation.
