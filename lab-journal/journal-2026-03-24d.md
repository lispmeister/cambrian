## Bugs and Fixes (Chronological)

### 1. Gen-0: Tests hit localhost instead of using in-process client

**Problem:** Test stage runs BEFORE start stage. Tests tried `requests.get("http://localhost:8401/health")` — but the server wasn't running yet.

**Fix:** Switched to `aiohttp_client` fixture from `pytest-aiohttp`, which creates an in-process test client. No running server needed.

**Lesson:** The Test Rig pipeline is build → test → start → health. Tests must be self-contained — they can't assume a running server. This is by design: tests verify code correctness, health check verifies the running container.

### 2. `aiohttp_client` fixture not found (twice)

**Problem:** `aiohttp_client` comes from `pytest-aiohttp`, NOT from `pytest-asyncio`. Missing from requirements.txt.

**Fix:** Added `pytest-aiohttp>=1.0` to requirements.txt.

**Lesson:** This happened twice — once for gen-0, once when running gen-1 tests on the host. The `pytest-aiohttp` dependency must be in every artifact's requirements.txt. The gen-1 spec prompt should make this clear, but it also needs to be in the Dockerfile's base packages (which it now is).

### 3. Accidentally committed viability-report.json and __pycache__

**Problem:** Test Rig writes `viability-report.json` to the artifact directory. `__pycache__/` gets created during pytest. Both got committed.

**Fix:** Added `.gitignore` to cambrian-artifacts (viability-report.json, __pycache__/, *.pyc, .pytest_cache/). Ran `git rm --cached` to remove already-tracked files.

**Lesson:** The artifacts repo .gitignore must exist before any pipeline runs.

### 4. Auto-promote called without git ops

**Problem:** `run_test_rig()` set `outcome="promoted"` in generations.json based on viability but never called `git_ops.promote()`. Branches were never merged.

**Fix:** Added actual `git_ops.promote()` / `git_ops.rollback()` calls after reading the viability report.

**Lesson:** The generation record lifecycle and git lifecycle must move in lockstep. Setting outcome in the record without performing the git operation is a consistency bug.

### 5. git_ops.promote() tried to create branch that already existed

**Problem:** `spawn_handler` creates the gen-N branch and commits the artifact. `promote()` then tried `git checkout -b gen-N` — which fails because the branch already exists.

**Fix:** Removed branch creation from `promote()`. It only needs: checkout main → merge gen-N → tag → delete branch.

**Lesson:** The git operation ownership is: spawn creates the branch, promote merges it. Don't duplicate responsibilities.

### 6. Gen-1: `python src/prime.py` fails (relative imports)

**Problem:** Running `python src/prime.py` treats it as a standalone script. `from . import generate` fails because there's no package context.

**Fix:** Added `src/__init__.py` and changed `entry.start` to `python -m src.prime`.

**Lesson:** Any Python artifact that uses intra-package imports MUST be run with `python -m`. This must be reflected in manifest.json `entry.start` and in the Test Rig's start stage.

### 7. Linter reverted entry.start

**Problem:** A pre-commit hook (linter) changed `"python -m src.prime"` back to `"python src/prime.py"` in `manifest.py`. The commit went through with the wrong value.

**Fix:** Re-applied the fix to both `src/manifest.py` and `manifest.json`.

**Lesson:** Watch for pre-commit hooks silently reverting intentional changes. Always verify the committed content matches intent, especially for string literals that look like they might be "wrong" to a formatter.

### 8. Test assertion mismatch after entry.start fix

**Problem:** `test_manifest.py` still asserted `python src/prime.py` instead of `python -m src.prime`.

**Fix:** Updated the test assertion.

**Lesson:** When fixing a value that appears in both code and tests, update both in the same commit.

## Architecture Insights

### Clean generation (genotype/phenotype split)

Gen-2 will be generated entirely from the spec (genome). The LLM prompt for Gen-2 contains:
1. The full spec (CAMBRIAN-SPEC-005)
2. Generation history (outcome records for gen-0, gen-1)
3. Generation number and parent

It does NOT contain Gen-1's source code. This is critical — Gen-2 is an independent interpretation of the spec, not a derivative of Gen-1. This validates the genotype/phenotype split: the spec is the genome, the code is the phenotype, regenerated from scratch each generation.

### Test Rig as invariant

The Test Rig is baked into the Docker image. It is never generated by the LLM. This makes it the fixed reference frame — it doesn't evolve, so it can objectively judge whether generated code meets the contract.

### Shared volume convention

The artifact directory is mounted at `/workspace` inside containers. The Test Rig reads from `/workspace`, writes `viability-report.json` to `/workspace`. The Supervisor reads the report after the container exits. No network protocol needed for result handoff.

### Exponential backoff in supervisor client

Gen-1's supervisor client uses exponential backoff (1s → 60s ceiling) for connection errors. This is important because the Supervisor might not be ready when Prime starts, and network errors during polling shouldn't cause immediate failure.

## Docker Run Command for Stage 2

To start Gen-1 and have it autonomously produce Gen-2:

```bash
docker run --rm \
  -e ANTHROPIC_API_KEY="$(cat .env | grep ANTHROPIC_API_KEY | cut -d= -f2)" \
  -e SUPERVISOR_URL="http://host.docker.internal:8400" \
  -e GENERATION=1 \
  -e CAMBRIAN_MODEL="claude-sonnet-4-6" \
  -v /Users/fix/projects/claude-code/cambrian-artifacts/gen-1:/workspace \
  --add-host host.docker.internal:host-gateway \
  cambrian-base
```

Note: `host.docker.internal` allows the container to reach the Supervisor on the host.
