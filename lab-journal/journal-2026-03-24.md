## Background

The Hyperagents paper (Zhang et al., 2026, arXiv:2603.19461) introduces metacognitive self-modification: splitting an agent into a task agent (solves the target task) and a meta agent (modifies both the task agent and itself). The critical innovation over DGM (Darwin Gödel Machine) is that the meta-level modification procedure is itself editable and subject to evolution — enabling what the authors call metacognitive self-modification.

This journal entry records how those ideas map onto Cambrian, what to adopt, and when.

---

## Mapping: Hyperagents to Cambrian

| Hyperagents | Cambrian |
|---|---|
| Task agent | Prime (reads spec, calls LLM, produces code) |
| Evaluation / fitness function | Test Rig (mechanical, binary, no LLM) |
| Meta agent | **Missing** — this is the gap |
| Population archive | Partial: `generations.json` + `gen-N-failed` git tags |

The mapping is clean. Cambrian's architecture was independently designed to have the same separation of concerns. The gap is real: in Hyperagents, the meta agent lives inside the organism and is subject to evolution. In Cambrian M1, the Supervisor manages everything infrastructure-related from the host side, and Prime has no self-improvement capability whatsoever (by design — M1 is reproduction only).

The Hyperagents insight sharpens what M2 should be:

> The process by which mutations are generated and evaluated should also be evolvable.

If mutation strategy lives in the Supervisor (host infrastructure), it's frozen. If it lives in the genome spec, it co-evolves with the task-solving logic. The spec-is-the-genome philosophy makes this the natural choice.

---

## What Cambrian Already Has (Proto-Hyperagents Infrastructure)

Several M1 mechanisms already lay the groundwork:

**Fitness vector** (15 metrics across Speed, Correctness, Economy, Robustness): Collected by the Test Rig from Gen-1 onward. Purely informational in M1 (viability stays binary). Exists so historical data is available when M2 arrives.

**Generation records** (append-only `generations.json`): Every generation attempt — successful or not — is recorded. Failed generations are git-tagged `gen-N-failed`. This is a passive population archive.

**Informed retry**: Failed code + structured diagnostics fed back into the LLM's context. This is Lamarckian learning within a generation. It's the closest M1 analog to a meta agent, but operates at the retry level, not the spec-mutation level.

**Verification contracts**: The organism declares its own fitness criteria; the Test Rig enforces them. Self-defined evaluation is part of the organism.

What's missing: Prime doesn't reason about its fitness history or propose spec mutations. That's intentional for M1, and will be added in M2.

## Design Decisions

### Decision 1: Mutation strategy lives in the genome, not the Supervisor

**Rationale:** Consistent with "the spec is the genome." If mutation strategy is hardcoded in the Supervisor, it cannot evolve — it's infrastructure, not organism. The Hyperagents paper shows that frozen meta-level strategies are the limiting factor in DGM's domain-coupling problem.

**Implication for M2:** SPEC-005 v2 will include a `## Meta-Strategy` section (see below). Prime regenerates both its task-solving code and its self-improvement code from that section. The Supervisor spawns and evaluates; Prime decides and proposes.

**Invariant preserved:** The Supervisor remains a dumb executor. It never runs organism-generated code on the host. All meta-agent logic executes inside sandboxed containers.

### Decision 2: Mutations are proposed, not applied

**Rationale:** This is already how Cambrian works at the phenotype level. Extending it to the genotype level: Prime generates a candidate mutated spec → Supervisor spawns a container with that spec → Test Rig evaluates → promote or discard. Prime never self-modifies in place.

This preserves the container-as-keep/revert-boundary invariant.

### Decision 3: Start with archive-based parent selection, not concurrent organisms

**Rationale:** Population-based evolution (multiple concurrent organisms, diversity-preserving selection) is a significant architectural change to the Supervisor. M2 should prove the mutation-strategy-in-genome concept first. The passive archive (generation records + git tags) already exists. Prime's meta-spec can include a strategy for choosing which past generation's spec to sample for mutation without requiring concurrent organisms.

Full population-based evolution is M3+.

### Decision 4: No M1 changes

**Rationale:** SPEC-004 is explicit: "Do not build M2 infrastructure during M1." The Hyperagents analysis strengthens the M2 design, but M1 is reproduction. M2 infrastructure would add complexity to a system that hasn't proven it can reproduce yet.

## Sketch: SPEC-005 v2 Meta-Strategy Section (M2)

When M2 begins, a new top-level section will be added to the genome spec. The section is structured for an LLM reader — it is executable behavior, not documentation.

```markdown
## Meta-Strategy (M2+)

This section defines how you improve yourself. It is part of your genome and
is subject to mutation and selection like everything else. When you generate
offspring, you generate a mutated version of this section along with everything
else.

### Mutation Generation

How you generate candidate mutations to your own spec.

**Current strategy:** After successfully promoting an offspring, inspect your
fitness vector and the last N generation records. Ask an LLM: "Given this
organism's fitness history, what is one change to the genome spec that would
likely improve fitness?" Produce exactly one candidate mutated spec. Submit it
for evaluation via the Supervisor's `/spawn` endpoint with `mutation: true`.

### Performance Memory

What you remember across generations about what worked.

**Current state:** Empty at genesis. As generations accumulate, record which
mutation categories (e.g., "reduce dependencies", "increase test coverage",
"simplify contracts") correlated with fitness improvements. Store this in
`mutation-log.json` alongside `generations.json`. This log is part of your
context when generating the next mutation.

### Selection Criteria

Beyond Test Rig viability, how you judge whether a mutation is worth exploring.

**Current criteria:** Viability only. When mutation-log.json has 10+ entries,
extend this to: prefer mutations in categories with a positive fitness-delta
history. Avoid repeating mutations that failed more than twice.

### Archive Sampling

When generating a mutation, you may use any past generation's spec as a
starting point, not only the current generation's. Consult `generations.json`
to identify past generations with high fitness scores. Sample from them when
the current lineage appears to have plateaued (no fitness improvement in the
last 5 promoted generations).
```

This section is intentionally simple at genesis. The expectation is that evolution will discover more sophisticated strategies. The `mutation-log.json` accumulates evidence; the LLM interprets it; and better interpreters will out-compete worse ones.

## Updated M2 Milestone Definition

**M2: Self-modify (Hyperagents-informed)**

Acceptance criteria:

1. Gen-N Prime (any N ≥ 1) produces a candidate mutated spec.
2. The mutated spec differs from the parent in at least one meaningful way (not whitespace or comments).
3. A new Prime is generated from the mutated spec.
4. The new Prime passes the Test Rig.
5. The new Prime's fitness vector shows improvement in at least one non-trivial metric relative to its parent.
6. The new Prime can reproduce (passes M1 acceptance criteria).
7. The mutation strategy itself (the Meta-Strategy section) has been preserved in the new spec — the organism has not discarded its self-improvement capability.

Criterion 7 is the Hyperagents-specific addition. It ensures that the meta agent is part of the organism and is transmitted across generations, not an external mechanism that gets lost.
